# WCAG 2.2 Level AA Transition — Radio-TTY Frontend

**Date:** 2026-06-06  
**Status:** Approved  
**Stack:** React 18 + TypeScript + MUI v9 + Vitest + @testing-library/react

---

## Background

The Radio-TTY frontend already targets WCAG 2.1 AA (color palettes, button sizing, aria-labels, aria-live regions). This spec closes the remaining gaps to reach full WCAG 2.2 Level AA compliance, including pre-existing 2.1 AA issues surfaced during audit.

### Audit Summary

| Criterion | Description | Status |
|-----------|-------------|--------|
| 2.4.11 Focus Not Obscured | TopBar is `position="static"`, no sticky overlays cover focused elements | Already met |
| 2.5.7 Dragging Movements | `DraggablePanel` drag-to-reorder has no keyboard/pointer alternative | **Must fix** |
| 2.5.8 Target Size (24×24px min) | All interactive targets ≥30px; IconButtons overridden to 44×44 | Already met |
| 3.2.6 Consistent Help | No help mechanism exists in the app | N/A |
| 3.3.7 Redundant Entry | No multi-step flows that re-ask for entered data | Already met |
| 3.3.8 Accessible Authentication | Username/password login, no CAPTCHA, `autoComplete` set | Already met |
| 4.1.3 Status Messages | MUI Snackbars lack explicit `aria-live` — unreliable in Safari + some ATs | **Must fix** |
| `window.prompt()` | Token substitution prompts use OS-native dialogs — inaccessible | **Must fix** |
| Automated a11y testing | No axe-core assertions in test suite — violations can regress silently | **Must add** |

Five fix-then-test units, executed in order. Each unit: fix the component, write the axe assertion for it, verify it passes, move on. Single implementation plan, single branch.

## Unit 1: DraggablePanel Keyboard Alternative (WCAG 2.5.7)

**File:** `frontend/src/components/DraggablePanel/DraggablePanel.tsx`  
**File:** `frontend/src/components/DesktopApp/DesktopApp.tsx`

### DraggablePanel.tsx changes

Add two optional props:

```ts
interface Props {
  id: string;
  children: React.ReactNode;
  onMoveUp?: () => void;
  onMoveDown?: () => void;
}
```

The drag handle `Box` gains:
- `role="button"` and `aria-label="Drag to reorder {id} panel"` so screen readers announce it as interactive
- Two MUI `IconButton` (`size="small"`) at the right end of the handle bar:
  - `aria-label="Move {id} panel up"`, `disabled` when `onMoveUp` is `undefined`
  - `aria-label="Move {id} panel down"`, `disabled` when `onMoveDown` is `undefined`
- Icons: `KeyboardArrowUpIcon` / `KeyboardArrowDownIcon` from `@mui/icons-material`

Both ▲ and ▼ buttons render together whenever **either** prop is passed (i.e. when the panel is inside a sortable list). Individual buttons are `disabled` at list boundaries (`onMoveUp` undefined → ▲ disabled; `onMoveDown` undefined → ▼ disabled). When **both** props are absent (standalone `DraggablePanel` use), no buttons render and the component is visually identical to today.

### DesktopApp.tsx changes

1. Add `KeyboardSensor` and `sortableKeyboardCoordinates` to the `DndContext` sensors:

```ts
const sensors = useSensors(
  useSensor(PointerSensor),
  useSensor(KeyboardSensor, { coordinateGetter: sortableKeyboardCoordinates }),
);
```

2. Extract a `handlePanelMove(fromIndex: number, toIndex: number)` function containing the `arrayMove` + localStorage + `save_user_prefs` logic currently inside `handlePanelDragEnd`. Both the drag end handler and the buttons delegate to this.

3. Each `DraggablePanel` in the sortable list receives:
- `onMoveUp={index > 0 ? () => handlePanelMove(index, index - 1) : undefined}`
- `onMoveDown={index < panelOrder.length - 1 ? () => handlePanelMove(index, index + 1) : undefined}`

### Test

`DraggablePanel.test.tsx`:
- Axe assertion: rendered with both callbacks provided (mid-list state)
- Axe assertion: rendered at index 0 (▲ disabled) and last index (▼ disabled)

## Unit 2: Snackbar Status Announcements (WCAG 4.1.3)

**File:** `frontend/src/components/DesktopApp/DesktopApp.tsx`  
**File:** `frontend/src/components/MobileApp/MobileApp.tsx`

MUI `Alert` renders `role="alert"` for all severities. However, the reliability of `role="alert"` on dynamically mounted elements varies across screen readers and Safari. Adding explicit `aria-live` attributes makes announcements consistent.

**Error snackbar `Alert`:** add `aria-live="assertive"` `aria-atomic="true"` — assertive because errors need immediate interruption.

**Success/publish snackbar `Alert`:** add `aria-live="polite"` `aria-atomic="true"` — polite because it's confirmatory.

Apply the same change to the equivalent Snackbars in `MobileApp.tsx`.

### Test

No `DesktopApp.test.tsx` exists and the component requires ~40 mocked props. Instead, add a focused test in a new `frontend/src/components/DesktopApp/__tests__/Snackbar.a11y.test.tsx` that renders only the two `Snackbar`+`Alert` combinations (success and error) in isolation with a `ThemeProvider` wrapper and an axe assertion on each.

## Unit 3: TokenPromptDialog — Replace `window.prompt()` (WCAG 4.1.3 + general accessibility)

**New file:** `frontend/src/components/TokenPromptDialog/TokenPromptDialog.tsx`  
**New file:** `frontend/src/components/TokenPromptDialog/__tests__/TokenPromptDialog.test.tsx`  
**File:** `frontend/src/App.tsx`

### TokenPromptDialog component

```ts
interface Props {
  open: boolean;
  tokens: string[];
  originalText: string;
  onSubmit: (resolvedText: string) => void;
  onCancel: () => void;
}
```

Renders a MUI `Dialog`:
- `aria-labelledby` pointing to a `DialogTitle` with id `"token-prompt-title"`: "Fill in message placeholders"
- One MUI `TextField` per token, label `"Value for {token}"`, `autoFocus` on the first field
- `DialogActions`: **Cancel** button (calls `onCancel`) and **Send** button (resolves substitutions, calls `onSubmit`)
- Send is `disabled` until all fields are non-empty
- `onKeyDown` on the last field: `Enter` submits (same as Send)
- On submit: for each token, replace all occurrences of `{token}` in `originalText` with the field value; pass resolved string to `onSubmit`

### App.tsx changes

Replace the `prompt_token` case body:

```ts
// Before: sequential window.prompt() loop
// After:
case 'prompt_token':
  setPromptState({
    tokens: msg.tokens,
    originalText: msg.original_text,
    operator: msg.operator,
    callsign: msg.callsign,
    targetCall: msg.target_call,
    targetName: msg.target_name,
  });
  break;
```

Add state:
```ts
const [promptState, setPromptState] = useState<PromptState | null>(null);
```

Where `PromptState` is a local interface (or added to `appTypes.ts`):
```ts
interface PromptState {
  tokens: string[];
  originalText: string;
  operator: string;
  callsign: string;
  targetCall: string;
  targetName: string;
}
```

Add handlers:
```ts
function handleTokenSubmit(resolvedText: string) {
  if (!promptState) return;
  send({ type: 'tx_message', text: resolvedText, operator: promptState.operator,
         callsign: promptState.callsign, target_call: promptState.targetCall,
         target_name: promptState.targetName });
  setPromptState(null);
}
function handleTokenCancel() { setPromptState(null); }
```

Render `TokenPromptDialog` alongside the other dialogs:
```tsx
<TokenPromptDialog
  open={promptState !== null}
  tokens={promptState?.tokens ?? []}
  originalText={promptState?.originalText ?? ''}
  onSubmit={handleTokenSubmit}
  onCancel={handleTokenCancel}
/>
```

### Test

`TokenPromptDialog.test.tsx`:
- Renders `open={true}` with two tokens; axe assertion
- Send button disabled when a field is empty; enabled when all filled
- Submit calls `onSubmit` with correctly substituted text
- Cancel calls `onCancel`

## Unit 4: axe-core Setup & Component Test Coverage

**Package:** `jest-axe` (dev dependency) — jest-compatible, works with Vitest globals, provides `axe()` and `toHaveNoViolations`.

### Installation

```bash
npm install --save-dev jest-axe
```

`jest-axe` v8+ ships its own TypeScript types — no separate `@types/jest-axe` package needed.

### Setup — `frontend/src/test/setup.ts`

Add at the bottom:
```ts
import { toHaveNoViolations } from 'jest-axe';
expect.extend(toHaveNoViolations);
```

### Vitest config — `frontend/vitest.config.ts`

Add `testTimeout: 10000` to give axe room on CI (axe DOM traversal can exceed the default 5s on complex renders).

### Axe assertions added to existing test files

Each assertion wraps the component in the app's `ThemeProvider` (light mode) to ensure theme-driven contrast is evaluated:

```ts
it('has no accessibility violations', async () => {
  const { container } = render(
    <ThemeProvider theme={makeTheme(false)}>
      <ComponentUnderTest {...minimalProps} />
    </ThemeProvider>
  );
  expect(await axe(container)).toHaveNoViolations();
});
```
